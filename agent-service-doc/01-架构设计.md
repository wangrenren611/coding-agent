## 1. 设计目标

### 1.1 核心目标

本架构设计旨在构建一个**生产级别**的 Agent 服务端应用，具备以下核心特性：

| 目标 | 描述 |
|------|------|
| **高可用** | 支持水平扩展，单点故障不影响服务 |
| **高性能** | 低延迟响应，支持高并发请求 |
| **一致性** | 强一致性的会话管理，保证数据不丢失 |
| **安全性** | 多层安全防护，完整的权限控制 |
| **可观测** | 全链路追踪，完善的监控告警 |
| **可扩展** | 模块化设计，支持功能扩展 |

### 1.2 非功能需求

| 需求 | 指标 |
|------|------|
| **可用性** | 99.9% (年度停机 < 8.76 小时) |
| **延迟** | P99 < 500ms (不含 LLM 调用) |
| **吞吐量** | 支持 1000+ 并发会话 |
| **扩展性** | 支持自动水平扩展 |

## 3. 技术选型

### 3.1 技术栈总览

| 层级 | 技术选型 | 版本 | 理由 |
|------|----------|------|------|
| **运行时** | Node.js | ≥20.0.0 | 高性能、事件驱动、生态丰富 |
| **语言** | TypeScript | ≥5.3 | 类型安全、生产力 |
| **Web 框架** | Fastify | ≥4.0 | 高性能、内置类型、插件化 |
| **API 文档** | @fastify/swagger | 最新 | 快速集成、OpenAPI 3.0 |
| **验证** | Zod + fastify-type-provider-zod | 最新 | 类型安全、声明式 |
| **认证** | @fastify/jwt + @fastify/cookie | 最新 | JWT 支持、会话管理 |
| **限流** | @fastify/rate-limit | 最新 | 分布式限流支持 |
| **SSE** | @fastify/sse | 最新 | Server-Sent Events |
| **数据库** | PostgreSQL | ≥15 | 可靠、JSON 支持、分区 |
| **ORM** | Prisma | ≥5.0 | 类型安全、自动迁移 |
| **缓存/队列** | Redis | ≥7.0 | 高性能、分布式支持 |
| **日志** | Pino | ≥10.0 | 高性能、结构化日志 |
| **监控** | Prometheus + Grafana | 最新 | 行业标准 |
| **追踪** | OpenTelemetry | 最新 | 供应商中立 |
| **容器** | Docker + K8s | 最新 | 行业标准 |

### 3.2 技术选型理由

#### 3.2.1 为什么选择 Fastify 而非 Express？

| 特性 | Express | Fastify | 结论 |
|------|---------|---------|------|
| 性能 | 中 | 高 | Fastify 约 2-3 倍快 |
| 类型支持 | 需要额外配置 | 开箱即用 | Fastify 更好 |
| 插件系统 | 中 | 丰富 | Fastify 插件化更优雅 |
| 错误处理 | 中 | 更好的设计 | Fastify 更安全 |
| 生态 | 大 | 增长中 | Express 更大但 Fastify 够用 |

**结论**：对于企业级高并发服务，Fastify 的性能优势明显。

#### 3.2.2 为什么选择 Prisma 而非 TypeORM/Knex？

| 特性 | TypeORM | Knex | Prisma | 结论 |
|------|---------|------|--------|------|
| 类型安全 | 中 | 低 | **高** | Prisma 最佳 |
| 学习曲线 | 高 | 中 | 低 | Prisma 最易上手 |
| 迁移体验 | 中 | 高 | 高 | Prisma 很好 |
| 查询性能 | 中 | 高 | 高 | 相当 |
| 事务支持 | 好 | 好 | 好 | 相当 |

**结论**：Prisma 的类型安全和开发体验最适合 TypeScript 项目。

## 5. 核心模块设计概述

### 5.1 Session Service (会话管理)

**职责**：
- 会话生命周期管理
- 会话配置管理
- 状态追踪与计量

**关键设计**：
- 无状态设计，状态存储在 Redis + PostgreSQL
- 使用 Redis 缓存热点会话
- PostgreSQL 持久化完整数据

### 5.2 Context Manager (上下文管理)

**职责**：
- 消息存储与检索
- 上下文版本控制 (乐观锁)
- 上下文压缩

**关键设计**：
- MVCC 多版本并发控制
- Redis 缓存当前上下文
- 压缩使用 LLM 生成摘要

### 5.3 Message Queue (消息队列)

**职责**：
- 消息操作串行化
- 异步持久化
- 并发控制

**关键设计**：
- 每个会话独立的 Redis List 队列
- 保证单会话消息操作串行化
- 批量写入 PostgreSQL

### 5.4 Auth Service (认证授权)

**职责**：
- 多租户支持
- API Key 管理
- JWT 令牌管理
- 权限控制

**关键设计**：
- 租户隔离 (行级安全)
- API Key SHA256 哈希存储
- JWT 短期访问 + 长期刷新

### 5.5 Tool Executor (工具执行)

**职责**：
- 工具注册与发现
- 工具执行调度
- 资源限制与隔离

**关键设计**：
- 安全级别分级 (safe/limited/sandboxed/isolated)
- 危险操作使用隔离环境
- 超时与资源限制

### 5.6 Worker System (后台任务系统)

**职责**：
- 异步任务处理
- 长时任务执行
- 系统定时任务

**关键设计**：
- 基于 Redis 的任务队列
- Worker 进程独立部署
- 支持断点续传
- 健康检查与心跳机制

> 详细设计见 [`11-后台任务系统设计.md`](./11-后台任务系统设计.md) 和 [`12-Worker基础知识详解.md`](./12-Worker基础知识详解.md)

## 7. 安全性设计

### 7.1 安全层次

| 层级 | 机制 | 描述 |
|------|------|------|
| **网络层** | VPC/安全组 | 网络隔离 |
| **传输层** | TLS 1.3 | 加密传输 |
| **应用层** | 认证 + 授权 | 身份验证 + 权限控制 |
| **数据层** | 加密存储 | 敏感数据加密 |
| **审计层** | 完整日志 | 操作审计 |

### 7.2 认证方式

| 方式 | 适用场景 | 有效期 |
|------|----------|--------|
| API Key | 服务间调用 | 长期 |
| JWT Access | 前端调用 | 15分钟 |
| JWT Refresh | 刷新 Token | 7天 |

### 13.2 K8s 金丝雀配置

```yaml
# 金丝雀部署
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: agent-api
spec:
  replicas: 10
  strategy:
    canary:
      steps:
        - setWeight: 5      # 5% 流量
        - pause: {duration: 10m}
        - setWeight: 25     # 25% 流量
        - pause: {duration: 5m}
        - setWeight: 50     # 50% 流量
        - pause: {duration: 5m}
        - setWeight: 100    # 100% 流量
      analysis:
        templates:
          - templateName: success-rate
        startingStep: 1
        args:
          - name: service-name
            value: agent-api
```

## 14. 灾难恢复方案

### 14.1 恢复目标

| 指标 | 目标 | 描述 |
|------|------|------|
| **RTO** | < 15 分钟 | 恢复时间目标 |
| **RPO** | < 1 分钟 | 恢复点目标 (数据丢失) |
| **MTTR** | < 30 分钟 | 平均恢复时间 |

### 14.2 灾难场景与恢复

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          灾难恢复场景                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  场景 1: 单 Pod 故障                                                       │
│  ─────────────────────                                                      │
│  棢测: K8s 健康检查失败                                                    │
│  恢复: K8s 自动重启 Pod (< 30s)                                            │
│  影响: 无 (其他 Pod 接管)                                                  │
│                                                                             │
│  场景 2: 节点故障                                                           │
│  ─────────────────────                                                      │
│  检测: K8s 节点 NotReady                                                   │
│  恢复: K8s 自动调度 Pod 到其他节点 (< 2min)                                │
│  影响: 正在执行的任务需要重试                                              │
│                                                                             │
│  场景 3: 可用区故障                                                        │
│  ─────────────────────                                                      │
│  检测: 监控系统告警                                                        │
│  恢复: 跨可用区部署自动接管 (< 5min)                                       │
│  影响: 需要确认数据同步                                                    │
│                                                                             │
│  场景 4: 数据库故障                                                        │
│  ─────────────────────                                                      │
│  检测: 数据库连接失败                                                      │
│  恢复: 主从切换 (< 2min)                                                   │
│  影响: 写入暂停，读取继续                                                  │
│                                                                             │
│  场景 5: 整个区域故障                                                      │
│  ─────────────────────                                                      │
│  检测: 监控系统告警                                                        │
│  恢复: 手动切换到备份区域 (< 15min)                                        │
│  影响: 需要 DNS 切换                                                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 14.3 备份策略

| 组件 | 备份频率 | 保留时间 | 恢复时间 |
|------|----------|----------|----------|
| PostgreSQL | 每小时增量 + 每日全量 | 30 天 | < 30 分钟 |
| Redis | AOF 持久化 | 不保留 (重建) | < 5 分钟 |
| S3 文件 | 跨区域复制 | 永久 | < 10 分钟 |
| 配置 | Git 版本控制 | 永久 | < 5 分钟 |
